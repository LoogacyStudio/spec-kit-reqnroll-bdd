# Reqnroll BDD Inject Tasks

## User Input

```text
$ARGUMENTS
```

## Objective

Inject a safe, traceable, idempotent BDD task block into the active Spec Kit `tasks.md`.

This command does not implement code. It only updates or proposes BDD implementation tasks for `/speckit.implement`.

## Precondition Rules

This command must only inject tasks when these files exist:

- `specs/{feature}/bdd-implementation-handoff.md`
- `specs/{feature}/bdd-traceability.md`
- `specs/{feature}/tasks.md`

If `bdd-implementation-handoff.md` is missing, skip without changing files.

If `bdd-traceability.md` is missing, skip without changing files.

If only `bdd-test-plan.md` exists, skip without changing files.

Do not infer task injection directly from `spec.md`, `plan.md`, or `bdd-test-plan.md`.

Do not create implementation tasks from scenario candidates alone.

This protects users who run `/speckit.implement` without using the Reqnroll BDD generate flow.

## Required Inputs

Read:

- `specs/{feature}/bdd-implementation-handoff.md`
- `specs/{feature}/bdd-traceability.md`
- `specs/{feature}/tasks.md` if present
- `specs/{feature}/spec.md` if present
- `.specify/memory/constitution.md` if present

If `$ARGUMENTS` names a feature folder, use that folder.

If multiple feature folders exist and the active feature cannot be inferred, ask the user to specify the target feature folder.

## Output File

Primary target:

```text
specs/{feature}/tasks.md
```

Fallback target if `tasks.md` cannot be safely updated:

```text
specs/{feature}/bdd-implementation-handoff.md
```

## Injection Rules

1. Preserve all existing non-reqnroll-bdd content in `tasks.md`.
2. Insert generated tasks inside a bounded marker block.
3. Re-running this command must replace the existing generated block instead of appending duplicates.
4. Every generated task must reference related Scenario IDs and AC IDs when available.
5. Generated tasks must guide `/speckit.implement`, not replace it.
6. Generated tasks must bind step definitions through the Application boundary.
7. Generated tasks must not instruct implementation through Godot Presentation nodes.
8. If the command cannot safely update `tasks.md`, it must write a proposed task block to `bdd-implementation-handoff.md` and clearly report the fallback.

## Marker Block

Use exactly these markers:

```markdown
<!-- reqnroll-bdd:tasks:start -->
<!-- reqnroll-bdd:tasks:end -->
```

## Required Inserted Structure

```markdown
<!-- reqnroll-bdd:tasks:start -->

## BDD Acceptance Test Tasks

> Generated by `speckit.reqnroll-bdd.inject-tasks`.
> Do not edit inside this block manually unless you are intentionally overriding generated BDD task guidance.

- [ ] BDD-001 Create or update the Reqnroll acceptance test project
  - Source: `bdd-implementation-handoff.md`
  - Boundary: Application
  - Notes: Keep the acceptance test project isolated from Godot Presentation runtime dependencies unless explicitly required.

- [ ] BDD-002 Include generated `.feature` files in the acceptance test project
  - Feature files: {feature_files}
  - Scenarios: {scenario_ids}
  - Criteria: {criteria_ids}

- [ ] BDD-003 Implement step definitions through the Application boundary
  - Suggested step classes: {suggested_step_classes}
  - Boundary: Application
  - Do not bind steps to Godot nodes, labels, buttons, signals, or scene tree structure.

- [ ] BDD-004 Add test support objects
  - Support: Scenario context, test data builders, in-memory repositories or fakes, Application-layer test facade if useful.

- [ ] BDD-005 Run acceptance tests
  - Command: `dotnet test`
  - Expected result: all Reqnroll acceptance tests pass.

<!-- reqnroll-bdd:tasks:end -->
```

## Task Generation Rules

Generate task content from:

- `bdd-implementation-handoff.md`
- `bdd-traceability.md`

Use:

- Scenario IDs from traceability.
- AC IDs from traceability.
- Feature file names from traceability or handoff.
- Suggested step classes from handoff.

If any value cannot be inferred, use a clear placeholder and add a warning below the block.

## Suggested Placement

Prefer appending the block near the end of `tasks.md`.

If `tasks.md` has a clearly marked implementation task section, insert the block after that section.

Do not reorder existing tasks.

## Fallback Behavior

If `tasks.md` does not exist:

1. Do not create a full `tasks.md` from scratch.
2. Append a `## Proposed BDD Task Injection` section to `bdd-implementation-handoff.md`.
3. Explain that task injection was skipped because `tasks.md` was missing.

If marker block is malformed:

1. Do not overwrite `tasks.md`.
2. Append a proposed task block to `bdd-implementation-handoff.md`.
3. Report that manual cleanup is required.

## Completion Criteria

The command is complete when:

- `tasks.md` contains exactly one `reqnroll-bdd` task block, or fallback output is written.
- Existing non-reqnroll-bdd tasks are preserved.
- Generated tasks reference AC IDs and SC IDs when available.
- Re-running the command does not duplicate generated tasks.
- The output clearly reports whether injection succeeded, updated an existing block, or fell back.
